Motivation & Core Goals

The entire project stems from a resourceful, secondary objective. The domain fsil.eu (meaning **Cloud** in Balti) was initially purchased *only* to host my private Nextcloud server at a subdomain (mycloud.fsil.eu). Finding the main domain empty, I recognized the wasted potential and initiated this portfolio project to utilize the existing asset fully and establish a professional presence.

This initiative provided crucial, hands-on practice in applying web development principles to a tangible, self-directed project, serving two primary strategic purposes:

  • Controlled Digital Identity: The website grants complete control over branding, aesthetic design, and content curation, ensuring a comprehensive and authentic representation of my skills and multi-faceted interests—a level of detail unattainable through standardized social profiles.
  • Practical Skill Demonstration: It functions as a dynamic portfolio, providing prospective employers with direct evidence of my practical coding, design, and project management capabilities, offering a deeper insight into my real-world competency and professional approach than a short interview or standard CV.

Technical Objectives

  • Implement a modern, mobile-first, fully responsive design using a utility-first CSS framework (Tailwind CSS) to ensure cross-device compatibility.
  • Structure the content logically into distinct verticals (IT, Chef, Influencer, Travel) while maintaining a cohesive brand identity.
  • Create a robust project template for detailed IT project write-ups, ensuring clear communication of technologies and methodologies.
  • Optimize for fast loading times and clear, intuitive navigation.

The Development Journey: A Chaptered Story

Chapter 1: Naming the Cloud and Building the Skeleton

The domain, fsil.eu, which means **cloud** in Balti, was initially secured specifically for my private Nextcloud installation (mycloud.fsil.eu). Once that primary goal was achieved, the decision was made to utilize the main domain name for this portfolio, transforming an empty placeholder into a functional professional site.

The development phase began in the **Linux Terminal** , where I methodically created the initial directory and file structure, building the robust skeleton that would hold the entire website.

Chapter 2: The Midnight Code Marathon (00:00 to 07:00)

After finishing a long evening shift at the restaurant at midnight, I sat down at my computer, fueled by pure creative energy. The goal was to translate my ideas into tangible structure as quickly as possible.

To save crucial time and accelerate development, I enlisted a powerful collaborator: the **Gemini API** . This allowed me to focus on high-level design and integration while rapidly generating and refining core code snippets.

By the time the sun rose at 7 AM, the essential structure, written in clean **HTML5** and initial **JavaScript** , was complete and functional.

Chapter 3: Defining the Aesthetic and Responsiveness

With the core files in place, the project shifted to visual design. I adopted **Tailwind CSS** for its utility-first approach. This allowed me to define a consistent, modern, and branded look across all pages using quick, clean class names.

The priority here was **mobile-first design** to ensure the multi-category content (IT, Chef, Influencer, Travel) adapted flawlessly to any device, guaranteeing a smooth user experience regardless of the screen size.

Chapter 4: The Linux Ghost Conflict (Cloudflared Tunnel vs. DNS Proxy)

The Need for the Tunnel

The initial problem was a critical infrastructure limitation: my ISP's cable router strictly blocked **port forwarding for HTTPS (port 443)**, which was essential for externally hosting my Nextcloud server. To securely expose this private service, I set up a **Cloudflared Tunnel** , establishing a secure, persistent, zero-trust connection to the Cloudflare edge without relying on opening any inbound ports on the router.

The Ghost Conflict

I deployed my portfolio website via **GitHub Pages** , configuring **Cloudflare** to proxy the public traffic to the required GitHub IP addresses. The site worked everywhere else, but my main **Linux** workstation couldn't load it.

The conflict arose because the local machine, upon resolving the public domain (`fsil.eu`), saw the existing local configuration for the *private* Cloudflared Tunnel. Instead of routing the request externally to the internet (which would then hit the public Cloudflare proxy), the traffic was incorrectly hijacked by the local tunnel's domain resolution logic, sending it on a loop.

The fix required advanced networking intervention: I had to adjust the **Docker** environment running the tunnel and, critically, manually edit my local network’s lookup tables in **/etc/hosts** to explicitly direct my workstation to the public GitHub Pages IPs, thereby forcing a public path for the portfolio domain and avoiding the local tunnel's ghost in the machine.

Chapter 5: The Living Portfolio (The Final Workflow)

The journey settled into a clean, efficient workflow. Now, every new idea, project detail, or life event begins as a local change. Once perfected, I push the updates to the **Git** repository, and the website automatically fetches and publishes the changes. This continuous deployment mechanism ensures that this website is never truly static; it is a live, ever-evolving reflection of my skills and experiences, always showing you something new and different as my life changes.

Updates & Key Challenges

Challenge 1: Navigation Relative Paths

A recurring issue was managing relative file paths in the navigation links. Since the main file (`index.html`) is in the root, and project details are in a subdirectory (`pages/`), all navigation links in the detail pages had to be prepended with `../` (e.g., `../index.html`).

Solution: Implemented a consistent path strategy across all files and tested navigation rigorously between subdirectories and the root.

Update 1: Sticky Header and Mobile Layout

The initial fixed header obscured content on small screens. The mobile menu logic was updated to use a `z-50` layer and include a subtle backdrop blur (`backdrop-blur-sm`) to ensure visibility without losing context of the page below.

Action: Refined CSS to ensure the sticky header shrinks slightly on scroll (future feature) and optimized padding to keep branding prominent but compact on mobile viewports.
